# Changelogs

本目录用于按日期归档更新日志：**一天一个文件**，文件名 `CHANGELOG_YYYY-MM-DD.md`。若同一天有多次变更，**写在当日同一文件内**：在 **「范围」** 中补列要点、在 **「代码变更」** 中顺延 **`### n)`** 小节；**不要**再单独起「追加：…」之类与正文平级的大标题块。

## 命名规则

- 文件名：`CHANGELOG_YYYY-MM-DD.md`
- 示例：`CHANGELOG_2026-05-07.md`

## 正文格式（固定写法）

以下与仓库内 [`CHANGELOG_2026-05-09.md`](CHANGELOG_2026-05-09.md)、[`CHANGELOG_2026-05-10.md`](CHANGELOG_2026-05-10.md)、[`CHANGELOG_2026-05-11.md`](CHANGELOG_2026-05-11.md)、[`CHANGELOG_2026-05-12.md`](CHANGELOG_2026-05-12.md) 等保持一致。

1. **一级标题**：`# 更新日志（YYYY-MM-DD）`（日期与文件名一致）。

2. **`## 范围`**  
   - 仅使用 **无序列表**（行首 `- `），一条一个业务主题，**不用** `1.` `2.` 编号。  
   - 语气：面向读者概括「做了什么 / 影响什么」，可含关键 API、路径、行为关键词。

3. **分隔线**：`范围` 与 `代码变更` 之间、`代码变更` 与 `验证记录` 之间各写一行 **`---`**。若存在 **`## 后续建议`**，其前也加 **`---`**。

4. **`## 代码变更`**  
   - 按主题拆成 **`### 1)`、`### 2)`、…**（半角数字 + 半角右括号），标题为简短动宾短语（可含反引号包裹的文件名或模块名）。  
   - 每个 `###` 下使用 **`- **标签**：`** 开头的列表项，常见标签：**`新建`**、**`修改`**、**`删除`**、**`影响文件`**、**`契约`**、**`文档`**、**`验证`**（与 05-09 部分小节一致）。  
   - 需要多行说明时：在对应 `-` 下用 **两个空格缩进** 再起 `-` 子列表（Markdown 嵌套列表）。  
   - 文件路径：使用 **相对仓库根** 的链接，如 `` [`frontend/src/foo.tsx`](frontend/src/foo.tsx) ``。

5. **`## 验证记录`**  
   - 无序列表；建议至少包含 **`- **自动化**：`**（如 `npm run build`、`npm run lint`、脚本命令）与 **`- **手工**：`**（可复现的操作与预期）。  
   - **不用** `1.` `2.` 编号。

6. **`## 后续建议`**（可选）  
   - 有未完成功能、已知技术债、产品待办时编写；若无则**整节省略**。

7. **禁止或避免**  
   - 避免与 **`## 范围` / `## 代码变更`** 平级的 **`## 追加：…`** 块；同日迭代一律并入上述结构。  
   - 避免在「范围」里写实现细节长代码块（细节放在「代码变更」）。

8. **索引**：每新增或显著修改当日文件后，在本文件 **「索引（按时间倒序）」** 顶部增加一行链到该日文件，摘要一句话（与 05-10 索引风格一致）。

## 模板（新建日志可复制）

```md
# 更新日志（YYYY-MM-DD）

## 范围

- **主题一**：用户可见变化或契约/文档要点（一句话到两三句）。
- **主题二**：…

---

## 代码变更

### 1) 第一块变更的短标题

- **修改**：[`path/to/file`](../path/to/file)  
  - 要点一。  
  - 要点二。

### 2) 第二块变更的短标题

- **新建**：[`path/new.tsx`](../path/new.tsx) — 一句话说明用途。
- **修改**：[`path/other.tsx`](../path/other.tsx) — …

---

## 验证记录

- **自动化**：`cd frontend && npm run build`、`npm run lint`（或当日实际命令）。
- **手工**：关键路径点选 / 接口冒烟与预期。

---

## 后续建议

- 可选：未完成功能或技术债一条。
```
